Platform Blueprint · v1 · Jul 2026

OpenNVR is an inference platform.
The NVR is just its first app.

The codebase has already drifted into a platform-and-apps shape — KAI-C is a real seam, the adapter contract is an ecosystem, and the camera agent is a standalone app. And examples/ is already 13 vertical apps with ~50 more on the roadmap. This blueprint makes those seams intentional, subtracts the NVR UI down to a world-class operator surface, and puts the governance story on screen.

02

Product surfaces: five apps in one shell

The first kind of app is a product surface — it has a UI and lives in the shell. Decompose these along interaction model and user, not features. Each has a genuinely different job, which is what makes it an app and not a tab. (There's a second kind entirely — the headless vertical detectors — in the next section.)

NVR

operator surface

Owns: Live wall, playback timeline, cameras, recordings, ONVIF/PTZ.

Why an app: dense, spatial, keyboard-driven, must be flawless. The control-room surface can't share a sidebar with firewall rules.

Camera Agent

conversational

Owns: Voice + chat, skills, standing alarms / monitors / reports / tasks.

Why an app: a conversational, always-on interaction model — nothing like the spatial NVR. Already standalone on :9100.

AI / Detections

ml-ops

Owns: Models, BYOM, adapter registry (KAI-C), detection results.

Why an app: different user (ML / ops) and lifecycle than cameras. This is where the adapter registry finally becomes visible.

Security / Network

near-separate product

Owns: Suricata alerts, firewall, NAT, network monitoring.

Why an app: this is essentially a network IDS. It does not belong next to "Live View."

Governance

the moat

Owns: Audit logs, compliance reports, sovereignty, BYOK.

Why an app: the regulated-buyer surface — your actual differentiator. Deserves to be seen, not buried in Settings.

Admin

shell infrastructure

Owns: Users, roles, settings, firmware, integrations.

Why not an app: shared by every app. Lives at the shell level, reachable from anywhere — not a peer of the NVR.

The gap: apps have no SDK, no catalog, no UI

The adapters got the full treatment — a contract, an SDK, a conformance suite, and (soon) a registry view. The vertical apps got 13 folders of ad-hoc Python, each re-implementing NATS subscription + state machine + alerting, configured by hand-editing YAML. Going from 13 → 60 that way is how an ecosystem becomes sprawl. Give the apps what the adapters have: an App SDK (a new detector in ~30 lines), an app catalog in the shell (install / enable / configure without touching YAML), and a per-app config + results surface so a "queue length" or "PPE compliance" app has somewhere to live.

Watch the convergence with the agent

The camera agent already ships monitor primitives — notify, count, line-crossing — that overlap the standalone examples (line-crossing, occupancy, and intrusion all redo work the agent's monitors also do, down to sharing ByteTrack). Decide the relationship deliberately: the examples should be the engine the agent orchestrates, not a parallel reimplementation of the same rules. One rule library, two front doors — a headless recipe and a conversational one.

The flagship is Live

Wire up the dnd-kit you already ship (drag-reorder tiles), per-tile WebRTC/HLS with graceful fallback, inline PTZ, and keyboard nav (1–9 focus, F fullscreen, space pause). This one screen decides whether people call your UI "best."

05

The Camera Agent, raised from demo to product

Good bones today — talk button, avatar, skills rail, alarm/monitor cards. As a first-class app it needs four things, all pointing at one idea: trust in a camera agent comes from showing what it saw.

01 Grounded conversation

Voice and text in one thread — with the frame and detections the agent reasoned over shown inline. Grounding is the differentiator; render it.

02 Skills as capabilities

Each skill with its backing adapter, availability, and a plain-English "what it answers." Keep runtime enable/disable — it genuinely helps small-model routing.

03 Standing intelligence

Alarms, monitors, reports, tasks get a real management surface — list, status, history, mute, edit. And persist state: an alarm that dies on restart is a liability.

04 Provenance

Every answer traceable to a frame, a model, a correlation ID. You already thread those through KAI-C — surface them. Same governance story, applied to the agent.

06

Make the crown jewel visible

A governed, sovereignty-enforcing adapter SDK is exactly what regulated and air-gapped buyers pay for — and it's the least visible thing in the product. Two moves.

A Build the registry view

One screen where an operator sees every installed adapter: model fingerprint, declared permissions, health, and its sovereignty verdict. This is the visible proof of your governance story.

B Enforce the contract in CI

You already have a conformance suite. Gate every adapter on it. As apps and third parties code against KAI-C, its contract becomes load-bearing — treat it like a public API.

Pick one adapter path now

There are two ways to ship an adapter — the SDK path and the legacy monolith — and the docs already say the monolith is "not recommended." Choose the SDK story publicly and deprecate the other before you have external adapter authors, or you'll support both forever and confuse everyone building on you.

These are cross-cutting and worth doing in parallel with decomposition — they're the actual gap between good and best. Adding apps on top of missing states just spreads the shallowness.

08

What to watch

Fragmentation

"More apps" done wrong = five inconsistent UIs. The shared shell + design system is what makes decomposition a strength. Build the shell first.

Two adapter paths

SDK vs legacy monolith. Pick one public story now, before external authors lock you into supporting both.

Depth debt

Current breadth is impressive but shallow. Fix state/design foundations in parallel, or new apps just spread the thinness.

KAI-C is load-bearing

As everything codes against it, its versioning, contract, and uptime become critical. Version it, document it, conformance-test it in CI.

09

Sequence

Order matters. The shell and foundations unblock everything downstream.

1

Shell + design system + P0 foundations

One shared layout, tokenized primitives, loading/empty/error states, error boundary, table virtualization. This is the platform floor. Status (Jul 2026): first slice shipped — shared UI primitives, react-query data layer, per-route code splitting. Still open: global error boundary, table virtualization.

2

Subtract the NVR

Cut the sidebar to Live / Playback / Cameras / Wall. Deepen Live into the flagship. Rehome everything else behind the app switcher — but sequence the switcher honestly: grouped sidebar sections deliver the IA split today (shipped Jul 2026); promote to a real app switcher only once AI and Governance have the depth to justify their own rails. A switcher onto shallow apps feels worse than grouped nav.

3

Registries + an app catalog + provenance

Put the governance story on screen — the adapter registry, an app catalog + App SDK for the vertical detectors, and the agent's frame/model/correlation trail. This is what turns 60 planned apps into a browsable ecosystem instead of sprawl. Status (Jul 2026): read-only adapter registry view shipped; permission approval, drift timeline, and metrics pending.

4

Promote the Camera Agent into the shell

Bring :9100 in as a first-class app: grounded conversation, capabilities view, persistent standing intelligence.
