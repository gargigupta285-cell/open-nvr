Engineering Design Spec companion to the platform blueprint

Give the apps what the adapters
already have: an SDK and a registry.

The adapter layer has a contract, an SDK, conformance tests, and a gateway. The vertical-detector layer — 13 shipped, ~50 planned — has 13 folders of copy-pasted Python wired by YAML. This spec factors the real duplication into an App SDK, closes the missing app-registry API, and puts a catalog UI on top — all against OpenNVR's actual NATS and KAI-C wire formats.

01

The duplication is measurable

Reading the 8 detector examples, the same code is pasted into each. The rule that makes a detector that detector is a small fraction of the file; the rest is boilerplate that belongs in a shared package.

The signal

~680 lines of the ~575–800-line detector files are boilerplate. loitering-detection.py minus the SDK-able parts is one handle_event method. That ratio is the whole argument for the SDK — and the reason a new detector should be ~40 lines, not a fork of an existing folder. Treat "575 → 40" as directional, not a target: real detectors keep their domain logic and land nearer 60–100 lines. The argument survives at 100; don't let the slogan drive API decisions.

04

What a detector becomes

The SDK folds config, alerts, geometry, the NATS loop, the CLI, signal handling, and the contract endpoints behind app(Detector).run(). What's left is the rule. Here's loitering-detection, whole.

loitering.py — the entire app~40 lines · was 575
from opennvr_app_sdk import Detector, Alert, app, AppManifest, Param, AlertType
from opennvr_app_sdk.geometry import bbox_center

MANIFEST = AppManifest(
    id="loitering-detection", name="Loitering Detection", version="1.0.0",
    category="perimeter", summary="Alerts when a watched object dwells in a zone.",
    requires_tasks=["object_detection"],          # checked vs GET /api/v1/adapters
    subscribes="opennvr.inference.>",
    params=[
        Param("watch_labels", list, default=["person"]),
        Param("threshold_s", float, default=30.0),
        Param("zones", "geometry.polygon", per_camera=True),  # drawn in the catalog UI
    ],
    emits=[AlertType("loitering", severity="high")],
)

class Loitering(Detector):
    manifest = MANIFEST

    def setup(self):
        self.dwell = self.keyed_state(ttl=self.cfg.threshold_s * 2)

    def on_detections(self, camera_id, dets, event):
        zone = self.zone_for(camera_id)                  # from validated per-camera config
        for d in dets:
            if d.label not in self.cfg.watch_labels: continue
            if not zone.contains(bbox_center(d.bbox, self.frame_size(camera_id))): continue
            st = self.dwell.touch((camera_id, d.label), at=event.completed_at)
            if st.age >= self.cfg.threshold_s and not st.alerted:
                st.alerted = True
                yield Alert(title="Loitering detected", camera_id=camera_id, severity="high",
                            correlation_id=event.correlation_id,
                            evidence={"dwell_s": round(st.age, 1), "label": d.label})

if __name__ == "__main__":
    app(Loitering).run()   # CLI · signals · NATS loop · /health · /manifest · self-register · dispatcher

05

The registry API that doesn't exist yet

OpenNVR has adapter and model endpoints but no app-registration API — the missing seam. Add one service and table, modeled on KAI-C's registry. The manifest is the schema that makes config safe and the UI generatable.

table: installed_apps
id            # "loitering-detection"
name          category      version
url           # http://loitering:9200
manifest_json # the AppManifest snapshot
config_json   # validated against manifest.params
enabled       status        last_seen
routes — /api/v1/apps
GET  /apps                 # catalog: installed + bundled index
POST /apps/register        # app calls this on boot
POST /apps/{id}/enable
POST /apps/{id}/disable
PUT  /apps/{id}/config     # validated vs manifest.params
GET  /apps/{id}/status     # proxies /health + /state
Why the manifest is load-bearing

Because params is typed and declarative, PUT /config can validate without app-specific code, and the catalog can render the config form without app-specific code. A "geometry.polygon" param becomes a zone editor on a camera still; a float becomes a slider. One schema, three consumers: the app, the API, the UI.

08

Rollout — prove it on one, then sweep

1

Extract the SDK

Promote alerts.py + zone.py + line.py + the loop/CLI boilerplate into opennvr-app-sdk (Apache-2.0, PyPI) — a sibling to opennvr-adapter-sdk. Add keyed_state, AppManifest, and the contract server.

2

Migrate loitering as the reference — plus one hard example per archetype

Rewrite loitering on the SDK (575 → ~40 lines). Green bar = its existing tests pass unchanged → parity proven before touching the rest. But loitering is the easy shape: before declaring the SDK stable, also validate keyed_state against package-delivery (per-track state machine) and FrameApp against license-plate-recognition (two-adapter chain) — otherwise the sweep in step 5 discovers the SDK only fits InferenceSubscriber and forces breaking changes mid-migration.

3

Add the app registry

installed_apps table + the six /api/v1/apps routes + health polling. RBAC-gated like every other resource.

4

Build the catalog view

Card gallery, manifest-driven config drawer, live dashboard — in the AI / Detections app, reusing the shell's design system.

5

Sweep the remaining detectors

Each shrinks to its rule. New roadmap apps (PPE, crowd, queue) start on the SDK — 40 lines, not a forked folder.

6

Re-point the agent's monitors

create_monitor instantiates SDK detectors. The duplication with the examples disappears; one rule library, two front doors.
