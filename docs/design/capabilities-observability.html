Engineering Design Spec third companion · capabilities & metrics

A capability is the atom.
Give it three doors and a gauge.

An adapter capability — object_detection, scene_caption, asr — is the shared unit under everything: a developer's API task, an operator's use case, and an agent's skill are all views onto the same atom. Today it's a bare string with no metadata, and the rich telemetry each adapter emits is scraped by no one. This spec adds the capability catalog (choose by intent) and the decision-grade metrics view (choose by evidence).

01

Two gaps, one already half-closed

The contract exploration turned up good news and a clear job. The metrics you need mostly already leave the adapters — nobody collects them. The capability metadata you need does not exist — it has to be added as a layer.

Gap A — capabilities are bare strings

tasks_advertised: list[str]. No label, description, category, tags, or use-case mapping. The examples README links task→use-case in prose only. So "choose by use case / task / dev" needs a taxonomy layer that doesn't exist yet.

Gap B — metrics are emitted, not surfaced

Every adapter's /metrics already exposes latency histograms, outcomes, inflight, and queue depth; every NATS event carries inference_ms + fingerprint. KAI-C polls only health. No scraper, no view. This is aggregation, not instrumentation.

03

The taxonomy layer, in two halves

Keep tasks_advertised as the free-text key it is. Add metadata in two places: a contract v1.1 descriptor so adapters can self-describe a task (dev + integrator doors), and a curated intent map that OpenNVR owns (the use-case door). Don't make adapters know use cases — that's a product-editorial concern, not the model's job.

contract v1.1 — optional, backward-compatible
# adapters MAY describe a task richly; tasks_advertised stays the fallback
class CapabilityDescriptor(BaseModel):
    task: str                        # "object_detection" — unchanged key
    label: str | None = None         # "Object Detection"
    summary: str | None = None
    categories: list[str] = []       # ["security","analytics"]
    tags: list[str] = []             # ["person","vehicle","perimeter"]
    example_result: dict | None = None

# CapabilitiesResponse gains one optional field:
    capabilities: list[CapabilityDescriptor] = []
use_case_map.yml — OpenNVR-curated
# product-owned, keyed on the task string
- use_case: "After-hours intrusion"
  intent: "Alert when a person or vehicle
           enters a zone outside hours."
  needs_capability: object_detection
  suggested_apps: [intrusion, loitering]
  suggested_adapters: [yolov8, yolov11]

- use_case: "Who's at the door?"
  needs_capability: face_recognition
  suggested_apps: [smart-doorbell]
  suggested_adapters: [insightface]
One schema, four consumers

The descriptor + intent map feed the catalog's use-case door, the task door, the dev door, and the agent's skill picker — the same four front doors the blueprint and App-SDK spec keep circling. Define the capability once; every surface reads it.

05

A thin collector on the poll you already run

KAI-C already polls every adapter every 60s for /health + /capabilities. Add /metrics to that same poll, and tap the NATS inference stream for per-event latency. Roll into a lightweight time-series — no full Prometheus required, though the /metrics format means you can bolt Prometheus + Grafana on later for free. Keep the rollup store deliberately dumb: bounded ring buffers and fixed windows, not a mini-TSDB growing inside KAI-C — long retention stays Prometheus's job.

GET /api/v1/adapters/{name}/metrics serves the rollups to the UI — the registry view the blueprint asked for, now with teeth

Prerequisite — the governed path must be the default

The NATS tap only sees inference that flows through POST /api/v1/infer/{adapter}. The OpenNVR backend still defaults to the legacy /infer route (OPENNVR_ADAPTER_CONTRACT in kai_c_service.py), which bypasses NATS, correlation IDs, and the audit trail. Flip that default to governed before or with step 2 of the rollout — otherwise the per-event half of the metrics (latency per camera, confidence, fingerprint) stays empty and the dashboard ships hollow.

07

Rollout

1

Ship the intent map + descriptor

Add the curated use_case_map.yml and the optional v1.1 CapabilityDescriptor. Backfill labels/tags for the ~9 tasks in use today — a half-day of editorial, no adapter changes required.

2

Add /metrics to the KAI-C poll

Scrape each adapter's Prometheus text on the existing 60s cycle; parse into rollups. Tap NATS for per-event inference_ms + confidence. Store windowed p50/p95/p99, rates, drift.

3

Expose the capability catalog

The three-door view in the AI / Detections app — use case, task, developer — reading the descriptor + intent map, cross-checked against live /api/v1/adapters.

4

Expose the decision dashboard

The per-adapter panel above, plus a fleet overview. Every metric captioned with the decision it drives — not a raw Grafana dump.

5

Wire the feedback loop

Let the catalog use the metrics: when two adapters offer the same capability, rank them by live latency / error / cost so "choose a model" is evidence-based, not a guess.
